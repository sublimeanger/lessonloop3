### ⚠️ CRITICAL: Lovable + Database Migrations
Lovable is connected to Supabase and deploys frontend + edge functions automatically. However, **Lovable does NOT reliably apply Claude Code's SQL migrations**. When Claude Code creates new migrations:
1. Migrations exist in `supabase/migrations/` in the repo
2. Lovable may bundle some into its own deployment but frequently skips Claude Code migrations
3. **After ANY Claude Code session that creates migrations, Jamie must manually apply them via Supabase SQL Editor**
4. Always remind Jamie: "New migrations were created — these need applying to the live Supabase DB via SQL Editor or `supabase db push`"
5. After applying, run `NOTIFY pgrst, 'reload schema';` in SQL Editor
6. All migration SQL must be idempotent: use `IF NOT EXISTS`, `CREATE OR REPLACE`, `DROP ... IF EXISTS` before `CREATE`, and wrap `ADD CONSTRAINT` in `DO $$ BEGIN ... EXCEPTION WHEN duplicate_object THEN NULL; END $$;`
7. CHECK constraints on existing tables must use `NOT VALID` to avoid failing on legacy data

## Database
### Key Tables (70+ total)
**Auth & Orgs:**
- `organisations` — 62+ columns. Timezone (IANA validated), currency_code (ISO 4217 validated), subscription fields (protected by `protect_subscription_fields` trigger), branding, continuation settings, payment plan defaults.
- `org_memberships` — role ENUM (owner, admin, teacher, finance, parent). org_id immutability trigger. **NEVER reference `organisation_members` — the correct table is `org_memberships`.**
- `profiles` — FK to auth.users

**Academy:**
- `students` — soft-delete (archived). Guardian linking via `student_guardians`. Notes field restricted from parents at RPC level.
- `teachers` — dual linking: `teacher_user_id` (direct) and `teacher_id` → `teachers.user_id` (indirect). `is_lesson_teacher()` and `can_edit_lesson()` check both paths. `pay_rate_type` and `pay_rate_value` for payroll.
- `locations` — timezone per location, rooms with capacity. Unique name per org (case-insensitive). Primary location via `set_primary_location` RPC.
- `lessons` — CHECK(end_at > start_at NOT VALID). FK to teacher, location, room. `is_open_slot` flag. Cannot delete if invoiced (trigger guard).
- `lesson_participants` — `rate_minor` snapshot (captured at lesson creation, preserved on edit). `is_open_slot` auto-cleared on INSERT.
- `lesson_notes` — accessed via RPCs only (`get_lesson_notes_for_staff`, `get_parent_lesson_notes`). Column-level privacy: `teacher_private_notes` never exposed to parents.
- `recurrence_rules` — weekly recurrence with DST-safe generation via date-fns-tz. 200-lesson cap.

**Billing:**
- `billing_runs` — FK to invoices via `invoices.billing_run_id`. Delete RPC (`delete_billing_run`) with paid invoice guard.
- `invoices` — status lifecycle: draft → sent → paid → voided. DB trigger enforces valid transitions. `total_minor`, `paid_minor` in integer minor units. Payment plan support (`plan_enabled`, `installment_count`, `installment_frequency`).
- `invoice_items` — CHECK constraints (NOT VALID): unit_price_minor >= 0, amount_minor >= 0, quantity > 0.
- `payments` — unique constraint on Stripe payment intent ID. Three-layer idempotency.
- `refunds` — `voided_at` + `voided_by`. RLS: no INSERT/UPDATE/DELETE for authenticated users (service_role only via edge functions). `record_manual_refund` RPC for non-Stripe refunds.

**Continuation & Credits:**
- `terms` — start/end dates per org
- `term_continuation_runs` / `term_continuation_responses` — status lifecycle with server-side deadline enforcement. `recalc_continuation_summary` RPC.
- `make_up_credits` — `voided_at` column used throughout (available_credits view, all RPCs filter `WHERE voided_at IS NULL`). Consumption via FOR UPDATE SKIP LOCKED. Max per term enforced. Credits on active waitlist don't expire. `void_make_up_credit` RPC. Auto-issued via `trg_auto_credit` trigger on attendance.
- `make_up_waitlist` / `enrolment_waitlist` — duplicate prevention index. Mutual exclusion on booking. Atomic conversion RPCs. `void_credits_on_student_delete` trigger.

**Pipeline:**
- `leads` / `lead_students` / `lead_activities` / `lead_follow_ups` — lead pipeline. `convert_lead` RPC for atomic conversion.
- `booking_pages` / `booking_page_teachers` / `booking_page_instruments` — public booking.

**Messaging:**
- `internal_messages` — threaded conversations. `payment_notifications` INSERT restricted to service_role only.
- `notification_preferences` — marketing emails default to opt-out (GDPR compliant).

### RLS Patterns
Every table has RLS enabled. Key patterns:
1. **Org isolation:** all queries scoped by org_id via org_memberships lookup
2. **Role-based:** `is_org_admin()`, `is_org_staff()`, `is_lesson_teacher()`, `is_parent_of_student()`
3. **SECURITY DEFINER RPCs:** ALL must verify `auth.uid()` + org membership at the top. No exceptions.
4. **Never use `USING(true)` or `WITH CHECK(true)`** on sensitive tables
5. **Column-level privacy:** lesson notes use RPCs to exclude `teacher_private_notes` from parent responses
6. **Parent portal guardian chain:** auth.uid() → guardians.user_id → student_guardians.guardian_id → students
7. **All SECURITY DEFINER functions** callable from frontend MUST have `is_org_*` auth check as first statement
8. **PostgreSQL grants EXECUTE to PUBLIC by default** — always add either internal auth check or `REVOKE EXECUTE FROM authenticated`

- **Update product knowledge** by editing knowledge-base.ts without touching index.ts.
- **Tool usage guidance** and **CSV import + rate snapshot explanation** included in KB.

| `bulk-process-continuation` | Server-side continuation | JWT + admin | |
| `looopassist-chat` | AI copilot | JWT + org member | v2 streaming + KB |
| `send-bulk-message` | Org-wide messages | JWT + admin | escapeHtml. List-Unsubscribe. |

### Edge Function Conventions
1. **Auth:** verify JWT, check org membership, check role
2. **Rate limiting:** use `_shared/rate-limit.ts`
3. **Error handling:** always return appropriate HTTP codes (never 200 on failure)
4. **Financial amounts:** always from DB, never from client
5. **Email:** Resend API, 3x retry, escapeHtml(), domain `lessonloop.net`
6. **config.toml:** email link/cron functions MUST have `verify_jwt = false`
