## Database
### Key Tables (56+ total)
**Auth & Orgs:**
- `organisations` — 62 columns. Timezone (IANA validated), currency_code (ISO 4217 validated), subscription fields (protected by `protect_subscription_fields` trigger), branding.
- `organisation_members` — role ENUM (owner, admin, teacher, finance, parent). org_id immutability trigger.
- `profiles` — FK to auth.users
**Academy:**
- `students` — soft-delete (archived). Guardian linking via `student_guardians`. Notes field restricted from parents at RPC level.
- `teachers` — dual linking: `teacher_user_id` (direct) and `teacher_id` → `teachers.user_id` (indirect). `is_lesson_teacher()` and `can_edit_lesson()` check both paths.
- `locations` — timezone per location, rooms with capacity. Unique name per org (case-insensitive). Primary location via partial unique index.
- `lessons` — CHECK(end_at > start_at). FK to teacher, location, room. `is_open_slot` flag. Cannot delete if invoiced (trigger guard).
- `lesson_participants` — `rate_minor` snapshot (captured at lesson creation, preserved on edit). `is_open_slot` auto-cleared on INSERT.
- `recurrence_rules` — weekly recurrence with DST-safe generation via date-fns-tz. 200-lesson cap.
**Billing:**
- `billing_runs` — FK to invoices via `invoices.billing_run_id`. Delete RPC (`delete_billing_run`) with paid invoice guard.
- `invoices` — status lifecycle: draft → sent → paid → voided. DB trigger enforces valid transitions. `total_minor`, `paid_minor` in integer minor units.
- `invoice_items` — CHECK constraints: unit_price_minor >= 0, amount_minor >= 0, quantity > 0.
- `payments` — unique constraint on Stripe payment intent ID. Three-layer idempotency.
- `refunds` — `voided_at` + `voided_by`. RLS: no INSERT/UPDATE/DELETE for authenticated users (service_role only via edge functions).
**Continuation & Credits:**
- `terms` — start/end dates per org
- `term_continuation_runs` / `term_continuation_responses` — status lifecycle with server-side deadline enforcement
- `make_up_credits` — `voided_at` column used throughout (available_credits view, all RPCs filter `WHERE voided_at IS NULL`). Consumption via FOR UPDATE SKIP LOCKED. Max per term enforced. Credits on active waitlist don't expire.
- `make_up_waitlist` / `enrolment_waitlist` — duplicate prevention index. Mutual exclusion on booking. Atomic conversion RPC.
**Messaging:**
- `internal_messages` — threaded conversations. `payment_notifications` INSERT restricted to service_role only.
- `notification_preferences` — marketing emails default to opt-out (GDPR compliant).
### RLS Patterns
Every table has RLS enabled. Key patterns:
1. **Org isolation:** all queries scoped by org_id via organisation_members lookup
2. **Role-based:** `is_org_admin()`, `is_org_staff()`, `is_lesson_teacher()`, `is_parent_of_student()`
3. **SECURITY DEFINER RPCs:** ALL must verify `auth.uid()` + org membership at the top. No exceptions.
4. **Never use `USING(true)` or `WITH CHECK(true)`** on sensitive tables — this grants universal access.
5. **Column-level privacy:** lesson notes use RPCs (not direct table access) to exclude `teacher_private_notes` from parent responses.
### Triggers
Key triggers to be aware of:
- `protect_subscription_fields` — blocks client tampering of subscription columns
- `org_id_immutability` — prevents org_id changes on 8+ tables
- `check_teacher_limit` / `check_student_limit` — counts from teachers/students tables (not org_memberships) with FOR UPDATE lock
- `check_lesson_conflicts` — prevents teacher + room double-booking
- `trg_attendance_not_future` — blocks marking attendance >15min before lesson start
- `trg_prevent_past_open_slot` — blocks open slot creation in the past
- `cleanup_attendance_on_cancel` — removes attendance when lesson cancelled
- `auto_clear_open_slot` — clears is_open_slot when participant added
## Edge Functions (82)
### Critical Edge Functions
| Function | Purpose | Auth | Notes |
|----------|---------|------|-------|
| `create-billing-run` | Generate invoices for a period | JWT + admin role | Rate limited 5/60min. Date validation, dedup, orphan cleanup |
| `stripe-webhook` | Process Stripe events | Stripe signature | Returns 500 on DB failure (forces retry). Three-layer idempotency |
| `stripe-checkout` | Create payment session | JWT + authenticated | Amount from DB, never client |
| `stripe-process-refund` | Initiate refund | JWT + admin/finance | Pending row before Stripe API call. Status check (no void refunds) |
| `send-invoice-email` | Email invoice to payer | JWT + org member | All data fetched from DB (never trusts client). 3x retry with backoff. Status guard (no paid/void) |
| `invite-accept` | Process invitation acceptance | JWT | Checks teacher limit before creating membership |
| `continuation-respond` | Parent accept/decline | Token-based | verify_jwt = false in config.toml |
| `waitlist-respond` | Parent respond to offer | Token-based | verify_jwt = false in config.toml |
| `send-enrolment-offer` | Email waitlist offer | Cron/service | verify_jwt = false in config.toml |
### Edge Function Conventions
1. **Auth:** verify JWT, check org membership, check role
2. **Rate limiting:** use `_shared/rate-limit.ts` config
3. **Error handling:** always return appropriate HTTP codes (never 200 on failure)
4. **Financial amounts:** always from DB, never from client request body
5. **Email:** use Resend API with 3x retry and backoff
6. **config.toml:** functions called from email links or cron MUST have `verify_jwt = false`
Cron jobs: See docs/CRON_JOBS.md for all 6 required scheduled jobs.

## Frontend
### Key Conventions
- **Currency:** always `formatCurrencyMinor(amount, currency)` or `currencySymbol(currency)` — never hardcoded £/$
- **Timezone:** always org timezone via `fromZonedTime`/`toZonedTime` from date-fns-tz — never browser timezone for business logic
- **Forms:** Zod validation, `isSaving` double-click protection, loading states, success toasts
- **Destructive actions:** AlertDialog confirmation with consequences explained
- **Role gating:** `isOrgAdmin` for admin/owner, route guards, component-level checks
- **Touch targets:** minimum 44px on mobile
- **Empty states:** meaningful messages with CTAs
- **Error states:** retry buttons, clear messages

## Production Readiness Audits
All 26 features passed independent audits (March 2026). Reports in repo root: `audit-feature-{01-26}-*.md`
### Systemic Patterns to Maintain
These patterns were established during the audit. All future code must follow them:
1. **Every SECURITY DEFINER RPC** must verify `auth.uid()` + org membership at the top
2. **Every financial amount** sourced from DB, never from client
3. **Every currency display** uses org currency dynamically
4. **Every date boundary** uses org timezone
5. **Every RLS policy** uses explicit role checks (never `USING(true)`)
6. **Every edge function** validates inputs, checks auth, handles errors with appropriate HTTP codes
7. **Every webhook handler** returns 500 on DB failure (forces retry)
8. **`voided_at IS NULL`** filter on every make_up_credits query
9. **Lesson notes** accessed via RPCs only (column-level privacy)
10. **Bulk operations** use atomic RPCs (not client-side loops)
11. **Auth/RLS:** All `SECURITY DEFINER` RPCs callable from the frontend MUST have an `is_org_*` auth check as the FIRST statement in the function body. Functions only called by service role (webhooks, cron) should have `REVOKE EXECUTE FROM authenticated`.
12. **Auth/RLS:** Never reference `organisation_members` — the correct table is `org_memberships`.
13. **Auth/RLS:** PostgreSQL grants `EXECUTE` to `PUBLIC` by default on new functions. When creating new `SECURITY DEFINER` functions, always add either (a) an internal auth check, or (b) a `REVOKE EXECUTE FROM authenticated` statement.

## Stripe
- **Checkout:** embedded via PaymentDrawer (portal) or redirect (admin)
- **Webhooks:** signature verification, three-layer idempotency (event dedup table → app check → DB unique)
- **Connect:** destination charges. Refunds go through platform account (not connected account)
- **Subscriptions:** 5 tiers. Webhook handles subscription.updated, payment_failed, subscription.deleted
- **Customer search:** uses parameterised `customers.list()` (not string interpolation)

## Calendar Rules
Calendar: All datetime comparisons against timestamptz columns must use explicit UTC (Z suffix or .toISOString()). Never rely on runtime timezone being UTC.
Calendar: Calendar view range computation must use org timezone, not browser timezone. Use fromZonedTime/toZonedTime from date-fns-tz.
Calendar: When editing recurring lessons with "this_and_future", always update the recurrence_rules row to reflect schedule changes.
Email: All HTML email templates MUST use escapeHtml() from _shared/escape-html.ts for any interpolated value — even admin-controlled data (defence-in-depth).
Email: From addresses use lessonloop.net domain (never .app or other domains). Billing emails use billing@, notifications use notifications@, invites use noreply@.
Email: Non-transactional emails (reminders, marketing, make-up offers) must check isNotificationEnabled() before sending. Transactional emails (invoices, receipts, refunds) always send.
