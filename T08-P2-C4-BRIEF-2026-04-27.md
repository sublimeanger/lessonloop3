## Why this exists

T08-P2-C2 shipped `check_cron_health()` with three CASE branches that all flag
`last_run_at IS NULL` as `'A_stopped_firing'` critical. The walk doc and brief
both specified registration-window suppression — when a cron has been
registered for less than 2× its scheduled interval, suppress Class A — but
that suppression never made it into the SQL.

The result, today: every freshly-registered cron (all 16 right now, post
T08-P2-C1's mass re-registration) reads as critical until it has fired at
least once. The watchdog itself, which fires for the first time at 09:30 UTC
the day after deploy, would email the operator a false-positive
"`cron-health-watchdog-daily` stopped firing" alert on Day 1.

C4 fixes this with a minimal logic change.

---

## Pragmatic suppression policy (Path B5)

When `last_run_at IS NULL`, the cron has never fired since registration. We
don't have an authoritative "registered at" timestamp from pg_cron, so we
cannot accurately compute "is this within registration window or pathological
silence." Path B5 sidesteps this: **suppress Class A unconditionally when
`last_run_at IS NULL`.**

Trade-off: a cron that's registered but never fires (because pg_cron is
broken at the system level, or `cron.job.active` was flipped to false at
some point) will not raise Class A.

This trade-off is acceptable because:
1. The original Track 0.8 problem ("cron fires but auth fails") is
   Class B territory, not Class A.
2. `cron.job.active = false` is an explicit operator action and visible
   via simple inspection.
3. pg_cron-level system breakage would manifest in many other ways
   (`cron.job_run_details` empty across the board) — operator would see
   the absence of all daily emails as a meta-signal.
4. Once a cron fires once, it's in scope of Class A from then on (any
   subsequent staleness > 1.5× interval triggers correctly).

Class B logic is unchanged.

## Migration

Create `supabase/migrations/20260509100000_cron_health_class_a_suppression.sql`:

```sql
-- T08-P2-C4: Suppress Class A 'stopped_firing' when last_run_at IS NULL.
--
-- Rationale: pg_cron does not expose a reliable per-job registration
-- timestamp. The original C2 logic flagged any active cron with
-- last_run_at IS NULL as Class A critical, which produces false positives
-- for freshly-registered crons (all 16 jobs after T08-P2-C1's mass
-- re-registration) until each one fires at least once.
--
-- Path B5 (chat-Claude walk OQ): suppress Class A unconditionally when
-- last_run_at IS NULL. Class A only fires when run history exists AND
-- the most recent run is older than 1.5× the cron's expected interval.
--
-- Trade-off: a cron that registers but never fires (e.g., active = true
-- but pg_cron broken at the system level) will not raise Class A. This
-- is acceptable; the original Track 0.8 problem space is Class B
-- ("cron fires but auth/HTTP fails"), and Class B logic is unchanged.
--
-- Function signature, return columns, permissions, and comment are
-- identical to C2. Only the three CASE expressions change.

CREATE OR REPLACE FUNCTION public.check_cron_health()
RETURNS TABLE (
  jobname text,
  schedule text,
  active boolean,
  failure_class text,
  severity text,
  last_run_at timestamptz,
  expected_interval_seconds bigint,
  age_since_last_run_seconds bigint,
  http_failures_24h integer,
  http_total_24h integer,
  http_failure_rate numeric,
  evidence jsonb
)
LANGUAGE plpgsql
SECURITY DEFINER
SET search_path TO 'public', 'cron', 'net', 'pg_catalog'
AS $$
BEGIN
  RETURN QUERY
  WITH job_intervals AS (
    SELECT
      j.jobid,
      j.jobname,
      j.schedule,
      j.active,
      CASE
        WHEN j.schedule = '*/15 * * * *' THEN 30 * 60
        WHEN j.schedule = '0 * * * *' THEN 90 * 60
        WHEN j.schedule LIKE '%* * * *' AND j.schedule NOT LIKE '*/%' THEN 36 * 3600
        ELSE 36 * 3600
      END AS interval_seconds
    FROM cron.job j
  ),
  job_last_runs AS (
    SELECT
      jrd.jobid,
      max(jrd.start_time) AS last_run_at,
      (array_agg(jrd.return_message ORDER BY jrd.start_time DESC))[1] AS last_return_message
    FROM cron.job_run_details jrd
    GROUP BY jrd.jobid
  ),
  job_age AS (
    SELECT
      ji.jobid,
      ji.jobname,
      ji.schedule,
      ji.active,
      ji.interval_seconds,
      jlr.last_run_at,
      EXTRACT(EPOCH FROM (now() - jlr.last_run_at))::bigint AS age_since_last_run_seconds
    FROM job_intervals ji
    LEFT JOIN job_last_runs jlr ON jlr.jobid = ji.jobid
  ),
  http_24h AS (
    SELECT
      jrd.jobid,
      count(*) FILTER (WHERE nhr.status_code BETWEEN 200 AND 299) AS http_ok_24h,
      count(*) FILTER (WHERE nhr.status_code IS NULL OR nhr.status_code >= 400) AS http_fail_24h,
      count(*) AS http_total_24h
    FROM cron.job_run_details jrd
    LEFT JOIN net._http_response nhr
      ON jrd.return_message IS NOT NULL
     AND jrd.return_message ~ '^[0-9]+$'
     AND nhr.id = jrd.return_message::bigint
    WHERE jrd.start_time > now() - interval '24 hours'
    GROUP BY jrd.jobid
  )
  SELECT
    ja.jobname,
    ja.schedule,
    ja.active,
    -- failure_class: NULL means healthy.
    -- T08-P2-C4: Class A no longer fires when last_run_at IS NULL.
    CASE
      WHEN ja.active AND ja.last_run_at IS NOT NULL
       AND ja.age_since_last_run_seconds > (ja.interval_seconds * 1.5)::bigint
        THEN 'A_stopped_firing'
      WHEN COALESCE(h.http_fail_24h, 0) >= 5
        OR (COALESCE(h.http_total_24h, 0) >= 12 AND COALESCE(h.http_fail_24h, 0)::numeric / NULLIF(h.http_total_24h, 0) > 0.5)
        THEN 'B_http_failing'
      WHEN COALESCE(h.http_fail_24h, 0) BETWEEN 1 AND 4
        THEN 'B_http_failing'
      ELSE NULL
    END AS failure_class,
    -- severity: scaled by class.
    -- T08-P2-C4: Class A severity gated on last_run_at IS NOT NULL.
    CASE
      WHEN ja.active AND ja.last_run_at IS NOT NULL
       AND ja.age_since_last_run_seconds > (ja.interval_seconds * 1.5)::bigint
        THEN 'critical'
      WHEN COALESCE(h.http_fail_24h, 0) >= 5
        OR (COALESCE(h.http_total_24h, 0) >= 12 AND COALESCE(h.http_fail_24h, 0)::numeric / NULLIF(h.http_total_24h, 0) > 0.5)
        THEN 'critical'
      WHEN COALESCE(h.http_fail_24h, 0) BETWEEN 1 AND 4
        THEN 'warning'
      ELSE 'info'
    END AS severity,
    ja.last_run_at,
    ja.interval_seconds AS expected_interval_seconds,
    ja.age_since_last_run_seconds,
    COALESCE(h.http_fail_24h, 0)::integer AS http_failures_24h,
    COALESCE(h.http_total_24h, 0)::integer AS http_total_24h,
    CASE
      WHEN COALESCE(h.http_total_24h, 0) > 0
      THEN round(COALESCE(h.http_fail_24h, 0)::numeric / h.http_total_24h, 3)
      ELSE 0
    END AS http_failure_rate,
    jsonb_build_object(
      'last_run_at', ja.last_run_at,
      'age_seconds', ja.age_since_last_run_seconds,
      'interval_seconds', ja.interval_seconds,
      'http_ok_24h', COALESCE(h.http_ok_24h, 0),
      'http_fail_24h', COALESCE(h.http_fail_24h, 0),
      'http_total_24h', COALESCE(h.http_total_24h, 0)
    ) AS evidence
  FROM job_age ja
  LEFT JOIN http_24h h ON h.jobid = ja.jobid
  ORDER BY
    -- T08-P2-C4: ordering also gated on last_run_at IS NOT NULL.
    CASE
      WHEN ja.active AND ja.last_run_at IS NOT NULL
       AND ja.age_since_last_run_seconds > (ja.interval_seconds * 1.5)::bigint THEN 0
      WHEN COALESCE(h.http_fail_24h, 0) >= 5 THEN 1
      WHEN COALESCE(h.http_fail_24h, 0) BETWEEN 1 AND 4 THEN 2
      ELSE 3
    END,
    ja.jobname;
END;
$$;

REVOKE ALL ON FUNCTION public.check_cron_health() FROM PUBLIC;
GRANT EXECUTE ON FUNCTION public.check_cron_health() TO service_role;

COMMENT ON FUNCTION public.check_cron_health() IS
'Surveys all cron jobs and returns failure/health status. Class A = stopped firing (only when run history exists); Class B = HTTP failing. Used by cron-health-watchdog edge fn (T08-P2).';

NOTIFY pgrst, 'reload schema';
```

### Class A — stopped firing

`cron.job.active = true` AND `cron.job_run_details.last_run_at IS NOT NULL`
AND `last_run_at` is older than `1.5×` the cron's expected interval.

The `last_run_at IS NOT NULL` requirement is intentional — pg_cron does not
expose a reliable registration timestamp, so freshly-registered crons (where
`last_run_at IS NULL`) cannot be distinguished from pathological silence.

re-registration deploys. A cron that registers but never fires will not
raise Class A; this trade-off is acceptable because the original Track 0.8
problem space is Class B (cron fires but HTTP fails), and pg_cron-level
breakage manifests in many other ways.

Severity: **CRITICAL**. Emailed daily until resolved.
