## Why this exists

`send-lesson-reminders` is the only hourly cron in the system. T08-P1 verification on 26 April surfaced 4 occurrences of `pg_net` 5-second timeouts (NULL `status_code` in `net._http_response`) on its scheduled `0 * * * *` slot in a 24h window, while the SQL-layer `cron.job_run_details` showed all 24 firings as succeeded. The SQL-layer success is correct (`net.http_post` queues the request and returns instantly); the 5s ceiling is the `pg_net` HTTP-layer timeout.

**Root cause (verified by chat-Claude reading `supabase/functions/send-lesson-reminders/index.ts:45-215`):** the fn runs everything serially through nested loops. For each org → each lesson → each guardian, it:

1. Awaits a `message_log` SELECT (per lesson, dedup check)
2. Awaits an `isNotificationEnabled` SELECT (per guardian)
3. Awaits `logAndSend`, which itself awaits 3 sequential operations: `message_log` INSERT → Resend HTTP → `message_log` UPDATE
4. Awaits `sendPush`, which is an internal HTTP call to the `send-push` edge fn

For an org with 10 lessons and 2 guardians per lesson, that's roughly 20 × (1 SELECT + 1 SELECT + 1 INSERT + Resend HTTP ~200ms + 1 UPDATE + push HTTP ~100ms) all serialised. At p50 latencies the run completes in 4-6 seconds; at p99 latencies it crosses the 5s threshold and `pg_net` records a NULL response. The edge fn keeps running on Supabase's 60s budget so the work completes — but the cron observability layer registers a "timeout" each time.

This isn't a correctness bug today (work completes; users get reminders). It's an observability and scaling issue that will get worse as user volume grows.

T08-P2's watchdog now classifies this as Class B WARNING territory (1-4 transient failures in 24h). The fix removes the false-positive noise and gives the fn comfortable headroom.

## Fix shape — bounded parallelism

Three changes, all within `supabase/functions/send-lesson-reminders/index.ts`:

**Change 1: Pre-batch `isNotificationEnabled` per org.**
Before processing lessons, run one `notification_preferences` SELECT for all guardian `user_id`s in the org's lesson set. Cache as a `Map<user_id, boolean>`. Eliminates N per-guardian round-trips per run.

**Change 2: Parallelise per-lesson and per-guardian with bounded chunks.**
Replace the serial `for (const lesson of lessons)` and inner `for (const [, { guardian }] of guardianMap)` with chunked `Promise.all`. Chunk size 5 — keeps Resend's 10 req/s rate limit headroom (Resend's default is 10 req/s globally; 5 concurrent leaves us nominally at 5 req/s steady-state per fn invocation).

**Change 3: Fire-and-forget push notifications.**
Replace `await sendPush(...)` with a non-awaited call. Push failure should not block subsequent guardian email work (the existing inline comment at line 202 already says this — but the await chain still blocks). Wrap the call in `.catch()` to log push failures without surfacing them to the surrounding error handler.

Combined effect on the 10-lesson 20-guardian example: 20 sequential email sends (serial ~4-6s) → 4 chunks of 5 guardians (parallel ~1-1.5s). Conservative estimate puts the fn well under 2s on real workloads.

---

## What does NOT change

- **Org-level loop stays serial.** Most orgs have 0-2 active orgs with `reminder_lesson_enabled = true` today; the loop is bounded by `MAX_ORGS_PER_RUN = 50`. Parallelising at the org level would make the rate-limit accounting harder and offers no real performance gain.
- **Per-lesson dedup stays serial.** It's a cheap SELECT; the cost is dominated by the email-send work. Could pre-batch with `IN(lesson_id)` similar to Change 1, but the optimisation is marginal — leave it.
- **Resend retry/backoff.** `sendWithRetry` exists in `_shared/send-invoice-email-core.ts`. NOT integrated here. That's a separate decision (retry behaviour during parallelisation needs careful thought re: rate limits) and is out of scope for this fix.
- **Cron schedule, auth pattern, validation.** Untouched. Pattern C, T08-P2-C1 capture wrapper, all preserved.
- **Email body, push payload, dedup window (48h).** Untouched.

## Implementation

### Step 1: Add a new helper `chunked` near the existing `formatRelativeTime` / `escapeHtml` helpers area (top of helpers section, ~line 80)

```typescript
/**
 * Run an async iteratee against an array in bounded-concurrency chunks.
 * Each chunk awaits Promise.all before the next chunk starts, so peak
 * concurrency is `chunkSize`. Returns the flat array of results.
 */
async function runChunked<T, R>(
  items: T[],
  chunkSize: number,
  iteratee: (item: T) => Promise<R>,
): Promise<R[]> {
  const results: R[] = [];
  for (let i = 0; i < items.length; i += chunkSize) {
    const chunk = items.slice(i, i + chunkSize);
    const chunkResults = await Promise.all(chunk.map(iteratee));
    results.push(...chunkResults);
  }
  return results;
}

const GUARDIAN_CONCURRENCY = 5;
```

Place `GUARDIAN_CONCURRENCY` next to the existing `MAX_ORGS_PER_RUN` constant at the top of the file.

### Step 2: Pre-batch the notification preference query in `processOrg`

const allUserIds = new Set<string>();
for (const lesson of lessons || []) {

### Step 3: Replace the nested loops with chunked parallelism

Currently the structure (lines 129-215) is:

```
for (const lesson of lessons || []) {
  // dedup check (serial — keep)
  // build guardianMap (sync, fast)
  for (const [, { guardian, students }] of guardianMap) {
    // do work — THIS IS THE HOT PATH
  }
}
```

Refactor to:

```typescript
let sent = 0;
let push = 0;
const errors: string[] = [];

// Per-lesson loop stays serial: dedup check is per-lesson, cheap.
for (const lesson of lessons || []) {
  try {
    const cutoff48h = new Date(now.getTime() - 48 * 60 * 60 * 1000).toISOString();
    const { data: existing } = await supabase
      .from("message_log")
      .select("id")
      .eq("related_id", lesson.id)
      .eq("message_type", "lesson_reminder")
      .gte("created_at", cutoff48h)
      .limit(1);

    if (existing && existing.length > 0) continue;

    // Build guardian list for this lesson (sync, no awaits)
    const guardianMap = new Map<string, { guardian: Guardian; students: string[] }>();
    for (const participant of lesson.lesson_participants || []) {
      const student = participant.student;
      if (!student || student.deleted_at) continue;
      const studentName = `${student.first_name} ${student.last_name}`;
      for (const sg of student.student_guardians || []) {
        const guardian = sg.guardian;
        if (!guardian || !guardian.email || guardian.deleted_at) continue;
        if (guardianMap.has(guardian.id)) {
          guardianMap.get(guardian.id)!.students.push(studentName);
        } else {
          guardianMap.set(guardian.id, { guardian, students: [studentName] });
        }
      }
    }

    // Per-guardian work parallelised with bounded concurrency.
    const guardianEntries = Array.from(guardianMap.values());
    const guardianResults = await runChunked(
      guardianEntries,
      GUARDIAN_CONCURRENCY,
      async ({ guardian, students }) => {
        try {
          // Notification preference (in-memory lookup, no await)
          if (guardian.user_id) {
            const enabled = notifPrefMap.get(guardian.user_id) ?? true;
            if (!enabled) {
              console.log(`Guardian ${guardian.id} has lesson reminders disabled, skipping`);
              return { sent: 0, push: 0, error: null };
            }
          }

          const { subject, html } = buildEmail(org, lesson, guardian, students, tz);

          const emailSent = await logAndSend(supabase, resendApiKey, {
            orgId: org.id,
            orgName: org.name,
            subject,
            html,
            recipientEmail: guardian.email,
            recipientName: guardian.full_name,
            guardianId: guardian.id,
            relatedId: lesson.id,
            messageType: "lesson_reminder",
          });

          // Fire-and-forget push. Push failure must not block email delivery
          // and must not delay subsequent guardian work.
          let pushFired = 0;
          if (guardian.user_id) {
            pushFired = 1;
            sendPush(supabase, {
              userId: guardian.user_id,
              title: `Lesson reminder`,
              body: `${students[0]} has ${escapeHtml(lesson.title)} ${formatRelativeTime(lesson.start_at, tz)}`,
              data: { type: "lesson_reminder", lessonId: lesson.id },
            }).catch((pushErr) => {
              console.error(`Push notification failed for guardian ${guardian.id}:`, pushErr);
            });
          }

          return {
            sent: emailSent ? 1 : 0,
            push: pushFired,  // counts pushes attempted, not confirmed delivered
            error: null as string | null,
          };
        } catch (guardianErr: unknown) {
          const msg = guardianErr instanceof Error ? guardianErr.message : String(guardianErr);
          return {
            sent: 0,
            push: 0,
            error: `Lesson ${lesson.id}, guardian ${guardian.id}: ${msg}`,
          };
        }
      },
    );

    for (const r of guardianResults) {
      sent += r.sent;
      push += r.push;
      if (r.error) errors.push(r.error);
    }
  } catch (lessonErr: unknown) {
    const msg = lessonErr instanceof Error ? lessonErr.message : String(lessonErr);
    errors.push(`Lesson ${lesson.id}: ${msg}`);
  }
}

return { sent, push, errors };
```

### Step 4: Note on the `push` counter semantic shift

Before this change, `push++` only fired after `sendPush` resolved with `true`. After this change, `push++` fires when push is attempted (because we no longer await the result). The accounting moves from "pushes confirmed delivered" to "pushes fired". This is acceptable because:

- Push delivery confirmation isn't authoritative anyway (FCM/APNs only guarantee the push was queued, not received).
- The console.log on push failure still surfaces issues for debugging.
- The success number reported in the response was always approximate.

If exact "pushes attempted that didn't error" is needed, we can re-introduce that accounting in a follow-up. For now the simpler fired-count is fine.
