## Project context

**LessonLoop** (lessonloop.net) is a music-academy management SaaS
that Jamie McKaye is building (Lovable frontend on Supabase backend,
React + TypeScript, Stripe Connect for billing, Xero for accounting
sync, native iOS app published to the App Store via Capacitor). The
codebase lives at `github.com/sublimeanger/lessonloop3`. Production
database is a Supabase project (project ref `ximxgnkpcswbvfrkkmjq`)
accessed by Lovable agents. Jamie is non-technical; he does not run
psql, git, or any developer tooling. All technical work is delegated
across a chain of AI assistants.

The "production hardening" project is a multi-month sequence of
audit-driven fixes across 17 functional areas (Area 0 → Area 16,
roughly mapping to user journeys + cross-cutting tracks; the
roadmap also tracks an extended set Area 17–27 for second-pass
integrations and platform surfaces). Each area has its own audit
document at `docs/audits/2026-04-area-N-*.md` (or in
`docs/archive/billing-closure-2026-04/` for older closed work). The
audits surface HIGH / MED / LOW findings; fix batches ("Batch 1Z",
"Batch 2A" through "Batch 2E", etc.) close findings in clusters.

The launch bar is "every area walked, every HIGH closed, every MED
either closed or formally deferred to a Track 0.X candidate, every
LOW filed". No "good enough" allowed; no half-done batches; no
silent skips. The roadmap is non-negotiable scope. V2 changes
execution speed — not what gets shipped.

## Workflow

The project uses **Workflow V2** (`WORKFLOW_V2_FAST_HARDENING.md`,
which is the canonical operating model — read this in full once
when you first come into the project). The condensed shape:

1. **chat-Claude** (a Claude.ai conversation, run by Jamie):
   strategy, prompt authoring, triage, decision support. Cannot
   push to GitHub. Cannot run production SQL. Writes the
   paste-ready prompt that drives each batch.

2. **Claude Code (web)**: opens PRs from chat-Claude's prompts.
   Writes migrations and edge functions. Runs typecheck, build,
   lint as repo-level verification. Cannot apply migrations to
   production. Cannot deploy edge functions. Cannot verify
   production state.

3. **Lovable**: applies migrations to production, deploys edge
   functions, runs read-only verification queries, runs
   behavioural spot-checks where its tooling permits (impersonated
   JWT against RPCs is the standard substitute for a real browser
   walk; Lovable cannot drive a logged-in user agent through the
   real React UI).

4. **Jamie**: merges PRs in GitHub. Drives the staff UI for
   behaviour checks Lovable cannot. Resolves manual-decision items
   (e.g. the 7 CW-F2 row dispositions). Confirms each batch as
   complete in chat. Pastes prompts and pastes outputs back across
   the four surfaces.

The core discipline:

- **Each batch ships one PR** with a single migration (or one
  migration + one edge-function change, if they belong together),
  never more. A batch is the unit, a finding is not.
- **Batch confirmations are folded into the next batch's PR**
  (Lovable's status flips from "pending" to "confirmed" inside
  the doc updates of the subsequent batch's PR). This keeps PR
  count down and creates a clean audit trail. Tiny docs-only PRs
  like this one are the exception — used when a closure is clean
  and there is no immediate next batch on the same surface.
- **STATUS.md is the single canonical handoff document.** Read
  its `Next session handoff` block first; everything else flows
  from there. The `Active area` and `In flight` blocks fill in
  the wider state; the rest of the doc is reference.
- **The audit doc for the active area is the canonical
  specification** for what closes in each batch. Never invent
  fixes; always trace each fix to a finding in the audit doc.
- **The agent never claims production state is verified** —
  that is Lovable's job, and Jamie confirms in chat. The agent
  verifies what it can verify at the repo level: typecheck,
  build, lint, grep-based syntax sanity, file-count assertions,
  diff-empty checks for unchanged sections of multi-section
  migrations.
- **Halt early, halt often.** If a finding's intent is ambiguous
  after reading the code, if production state must be verified
  before continuing, if any irreversible action would be needed,
  if scope drifts beyond the batch's stated bounds — halt. The
  halt format (one chat message; STATUS handoff updated to
  describe the resume point) is in `WORKFLOW_V2_FAST_HARDENING.md`
  Rule 3.6.

## How chat-Claude should operate in this project

**Read STATUS.md first.** Every fresh chat-Claude session starts
by reading STATUS — specifically the `Next session handoff` block,
then the `Active area` block, then any open `Awaiting Jamie
decision` items. Skim `Recent decisions on file` for product
context. If the session is picking up after a halt, the handoff
block tells you what specifically the halt is waiting on; resume
there, do not re-plan from scratch.

**Never invent fixes.** Each fix in a batch must trace to a
finding in the active area's audit doc. If you find yourself
proposing a change that isn't in the audit, stop. Either the
audit needs amending (file a new finding, get Jamie's approval,
then proceed) or the change is out of scope.

**Sandbox-test SQL involving non-trivial Postgres features
before writing a prompt.** This is the workflow improvement that
came out of the Batch 1Z incident: chat-Claude wrote a CW-F4
trigger refactor that combined `INSERT OR UPDATE OR DELETE` with
`REFERENCING NEW TABLE / OLD TABLE` on a single trigger. Postgres
rejects this with error 0A000 ("transition tables cannot be
specified for triggers with more than one event") at apply time —
not at parse time. Claude Code's repo-level verification
(typecheck + build + lint) cannot catch a Postgres-only runtime
rejection. The migration shipped, Lovable applied, Postgres
rolled back, twice. The corrected re-apply (PR #379) split each
surface's combined trigger into three event-specific triggers.
The lesson: when a migration uses transition tables, exclusion
constraints, partitioning, GiST indexes, generated columns,
identity-column oddities, or any feature whose failure modes are
runtime-only, chat-Claude should run the SQL against a real
Postgres before writing the prompt. The grep-based migration
sanity step in V2 verification is necessary but insufficient.

**Scope drift is the single biggest risk.** When you spot a
related finding mid-batch, file it for a future batch — do not
expand the current one. Each batch should ship a tight, reviewable
diff with one clear closure scope. Jamie's batch approval is the
contract; expanding mid-batch breaks the contract.

**Batch confirmations fold into the next batch's PR.** Don't
write a tiny docs-only PR every time Lovable confirms apply.
Track Lovable's status in chat-Claude's working memory; flip the
STATUS line in the next batch's PR. Exceptions: when a batch
closes an area cleanly and there's no immediate next batch on
the same surface, a docs-only PR (like this 2026-05-02 one) is
the right shape — but those are rare.

**Behaviour-walking distinction (Path A vs Path B).** Path A is
a real-browser walk with a logged-in staff user; Path B is the
RPC + edge-function + DB-state layer using impersonated JWT
context (set `request.jwt.claims` so `auth.uid()` resolves
correctly inside SECURITY DEFINER functions). Path B is
structurally equivalent to Path A for everything below the React
render layer. Lovable's environment cannot do Path A — it cannot
drive a logged-in browser session — so it walks Path B and code-
reviews pure-rendering claims. Document which path you used.

## How Claude Code should operate in this project

**The V2 PR body shape is non-negotiable.** Every batch PR has
six H2 sections, in order: `## Summary`, `## Findings closed`,
`## Repo-level verification (done by agent)`,
`## Lovable after-merge actions (Jamie does these)` (with four
H3 subsections: Migrations to apply, Edge functions to deploy,
Production SQL verification, App behaviour checks),
`## Next session handoff update`, `## Roadmap impact`. Each
subsection has content or `(none)`. Never omit a subsection. The
PR body is structured because Jamie reads it as a checklist when
driving Lovable post-merge.

**Section diff verification on multi-section migrations.** When
a corrective re-apply ships a migration that re-uses byte-
identical sections from a prior broken migration, run `diff` on
each unchanged section and assert the diff is empty in the PR
body. This was the verification pattern that made PR #379's
review tight — Sections 1, 2, 4, 5 of `20260516110000_*.sql`
were byte-identical to PR #378's broken file and the PR body
showed empty-diff confirmation per section. Section 3 was the
only intentional change.

**Idempotency discipline.** Every migration must be re-runnable
against partial state without breaking. Use
`CREATE OR REPLACE FUNCTION`, `IF NOT EXISTS`,
`DROP TRIGGER IF EXISTS` before any `CREATE TRIGGER`,
count-guard `DO $$ BEGIN IF (SELECT COUNT(*) …) THEN … END IF;
END $$;` blocks for one-off data fixes. The Batch 1Z incident's
broken migration left production in a partial state (Sections
1/2/4/5 had partly applied before Section 3 failed and the
transaction rolled back); the corrective re-apply needed to
handle the case where some of those sections might already be
in place from a different rollback timing. Idempotent migrations
make this easy.

**Never apply migrations from Claude Code.** Even when you have
network access, the production-access boundary (Rule 0) means
Claude Code does not call the Supabase Management API, does not
verify production Supabase state by any means, does not deploy
edge functions. The only thing Claude Code does to production is
write code that Lovable later applies. This boundary is what
keeps the chain auditable.

**Never claim production state without Lovable confirming.** PR
bodies say "Repo-level verification (done by agent)" — that is
the agent's ceiling. Production verification (queries A–H,
behaviour spot-checks, post-apply observations) lives in the
"Lovable after-merge actions" section as instructions for Jamie
to drive Lovable through, not as claimed results. The handoff
block's `Lovable status` field has three valid values:
`pending until Jamie confirms`, `confirmed complete YYYY-MM-DD
HH:MM UTC`, or `N/A`. Claude Code only ever writes the first or
the third. The middle value is set by chat-Claude in the next
batch's PR after Jamie confirms in chat.

## How Lovable should operate in this project

**Read-only first.** Before applying any migration, run the
read-only diagnostic queries in the PR body. The diagnostic
state shapes whether the migration is the right next action. The
CW-F9 LL-2026-00008 follow-on (Session 6, this Batch 1Z arc) was
caught precisely because Lovable's post-PR-#379 verification ran
the strict ledger query and surfaced one drift row. Without that
read-only step, the residual would have lived in production
silently.

**Halt when uncertain.** Lovable's role is the production-access
surface, but it is not infallible. When a verification query
returns unexpected results (drift rows, type mismatches,
behavioural anomalies), surface those to chat-Claude rather than
working around them. Lovable's boundary-setting in the original
Batch 1Z incident was correct behaviour: it rolled back twice
and surfaced the PostgreSQL 0A000 error verbatim, which let
chat-Claude diagnose root cause and write the corrective re-
apply. If Lovable had silently retried with a workaround (e.g.
applying Sections 1/2/4/5 alone and dropping Section 3), the
audit trail would be muddier and the trigger refactor would have
shipped half-broken.

**Substitute Path B for Path A when needed, but document the
substitution.** When a behaviour check requires a real browser
session and Lovable can't drive one, run the equivalent test at
the SQL level under impersonated JWT context. The Batch 2B
make-up flow checks (cancel booked / decline offered / accept
offered) used this pattern: same RPC, same SECURITY DEFINER,
same triggers. Note the substitution in the verification log so
future readers know which layer was tested.

**Verification methodology — no-op UPDATE for trigger sanity.**
When testing that a statement-level trigger fires correctly on
UPDATE, the canonical pattern is `UPDATE table SET col = col
WHERE id = <pre-selected row>` — one trigger fire per surface,
no FK pain on test data, easily restorable via direct UPDATE on
the parent table that has no statement-level trigger of its own.
Codified during Session 6 of the Batch 1Z arc; the Step 4 sanity
test migration (`20260502060816_…_cw_f4_trigger_sanity_test.sql`)
is the worked example.

## Recurring failure modes to watch for

**Postgres-runtime issues invisible to repo-level verification.**
Transition table syntax, enum casts on missing values, exclusion
constraint conflicts, GENERATED column / GENERATED ALWAYS AS
IDENTITY mismatches, GiST/GIN index requirements on operator
classes — none of these surface at typecheck, build, lint, or
even `psql --syntax-check`. The Batch 1Z incident was the
canonical example: `INSERT OR UPDATE OR DELETE` combined with
`REFERENCING NEW TABLE / OLD TABLE` parses fine but apply-time
errors with 0A000. The corrective workflow improvement: chat-
Claude sandbox-tests SQL involving such features against a real
Postgres before writing the prompt; Claude Code's verification
remains the agent's ceiling.

**Process drift — assistant doing work that belongs to another
assistant in the chain.** During the original Batch 1Z incident,
chat-Claude started writing a corrective migration draft inline
in the chat, which strictly speaking should have gone to Claude
Code as a fresh prompt from the start. The drift was small (the
draft was correct) but it muddies the audit trail. Each
assistant's role is canonical; when in doubt, hand off to the
right surface even if it adds a paste step.

**Doc sync gaps after multi-session arcs.** STATUS.md often lags
the actual production state by a session or two — Session 5
shipped PR #379, Session 6 shipped two follow-on migrations and
updated the audit doc, but STATUS still said "Batch 1Z corrected
re-apply in flight" until this 2026-05-02 docs-only PR caught
it up. The pattern: after any multi-session batch arc, the next
session's first task is reconciling STATUS + POLISH_NOTES against
the actual repo state. The handover doc you are reading now is
explicitly the close-out of the Batch 1Z arc's doc sync.

**Lint baseline counting basis differs between Lovable and
Claude Code.** Lovable reports 1228 problems / 7 errors / 1221
warnings on the same codebase where Claude Code reports 258
errors-only. Same lint config; different counting basis (one
includes warnings, the other doesn't). PR bodies should pick one
counting basis and stick to it within the PR; cross-PR
comparisons need to translate. Filed in POLISH_NOTES under
process improvements; not a bug, just a canonical-baseline
ambiguity.

**`auth.uid() IS NULL` short-circuit pattern.** Defensive RPC
code that reads `IF auth.uid() IS NOT NULL AND NOT
is_org_finance_team(...) THEN RAISE` silently allows service-
role-backed callers to bypass authorization (CW-F3 in Area 1).
The correct pattern is `IF NOT is_org_finance_team(auth.uid(),
...) THEN RAISE` because `is_org_finance_team(NULL, ...)`
returns false. Worth grepping for the broken pattern in any RPC
that mutates org-scoped data.

**Per-row vs statement-level trigger amplification.** Per-row
firing on `pdf_rev` bumps produced N+1 audit_log rows on an
N-item invoice (CW-F4). Statement-level via transition tables
gives you exactly one bump per statement. Watch for any new
trigger that might compound at scale; default to statement-level
unless per-row is required by the trigger's semantics.

**Schema integrity gaps.** CW-F2 (no payer-XOR CHECK on
`invoices`) and CW-F12 (no enum/CHECK on
`invoice_installments.status`) are two cases where the codebase
implicitly assumed an integrity constraint the schema didn't
enforce. The pattern is dangerous because rule violations can
sit in the data for weeks before a query surfaces them. A
broader schema audit is filed for Track 0.X / future schema-
hardening batch — likely candidates: any other status columns,
any FK-like columns without actual FK constraints, any uniqueness
invariants enforced only in code.
